# khan v1 — design

## Context

User has a paid SideNotes subscription and wants to replace it with a self-built helper named **khan** that keeps every SideNotes feature and adds:

1. **Global CLI notification bridge.** Any process (Claude Code post-run hook, cron, scripts, memos) calls `khan notify ...` to push a notification to the running app. Two display modes: **banner** (auto-dismiss, like NSUserNotification) and **fix** (persistent, pinned to notch or sidebar).
2. **Cross-device push** between Mac, iPhone, iPad on the same iCloud account (`khan notify --to=<device>` / `--all-devices`).
3. **Minute-level iCloud sync** of notes and inbox messages.
4. **Notch UI on Mac**: idle state = tiny pill near the notch; click expands; "open in window" gives a normal main window. Plus a SideNotes-style edge sidebar with Open Bar + Hot Side + pin/Fix mode.
5. **iOS/iPadOS app** sharing one CloudKit container with the Mac app.

User-confirmed decisions:
- Distribution: **Mac App Store** (sandboxed) + **iOS App Store**.
- Stack: **Swift + SwiftUI**, native.
- Notch: **DynamicNotchKit** (MIT-friendly, MrKai77).
- One-shot v1 design (no MVP staging).
- khan does **not** execute arbitrary shell — it's a notification/notes sink only; external tools push in via CLI.
- CLI is bundled inside the app; first-launch wizard offers to install/symlink to `/usr/local/bin/khan`.

Repo is greenfield (only `LICENSE`, `README.md`, single initial commit).

## Approach

A single Xcode workspace with macOS + iOS app targets, share/widget/intent extensions, a Swift-package shared core (`KhanCore`/`KhanUI`/`KhanMacChrome`), and a `khan` CLI Swift Package. All targets share **one App Group** (`group.com.gavin.khan.shared`) and **one CloudKit container** (`iCloud.com.gavin.khan`).

The Xcode project is generated by **XcodeGen** from `project.yml` checked into the repo so the multi-target setup is reproducible and reviewable. (Alternative: hand-built `.xcodeproj`. Rejected — too much friction for multi-target greenfield.)

### 1. Targets

- `Khan-macOS` (sandboxed app, MAS)
- `Khan-iOS` (iOS/iPadOS 17+ app)
- `KhanShareExtension-macOS` / `KhanShareExtension-iOS`
- `KhanWidget-macOS` / `KhanWidget-iOS` (WidgetKit)
- `KhanIntents` (App Intents — Add Note, Push Notification, Mark Done, Open Sidebar)
- `khan` CLI (Swift Package, `ArgumentParser`) — built once, copied into `Khan.app/Contents/Resources/khan` as a build phase

### 2. Data model (SwiftData `@Model` in `KhanCore`)

`Folder`, `Note`, `ChecklistItem`, `Tag`, `Attachment`, `Message`, `Device`, `UserSettings`. Notes and Messages are **separate models** (different lifecycles/UI) but share `Tag` + `Attachment`. Messages are append-only except `state`/`snoozedUntil`; `Message.promotedNote` lets a user convert a notification into a permanent note.

CloudKit zones (all in private DB):
- `NotesZone` — Folder/Note/ChecklistItem/Tag
- `MessagesZone` — Message + message attachments
- `OutboxZone` — pending cross-device pushes (raw CKRecord, NOT SwiftData-managed; cleaned up after 24 h)
- `DevicesZone` — one Device record per device

Attachments stored as files at `<AppGroup>/Attachments/<uuid>.<ext>`; mirrored as `CKAsset` siblings in CloudKit. SwiftData only holds metadata + relative path.

### 3. iCloud sync

SwiftData with CloudKit auto-sync via `ModelConfiguration(cloudKitDatabase: .private("iCloud.com.gavin.khan"))`. On top of that:
- A 60-second `SyncTimer` actor flushes + fetches change tokens to honor "minute-level sync".
- Manual "Sync now" menu item + `khan sync` CLI subcommand.
- On Mac, also re-poke on `NSWorkspace.didWakeNotification`. On iOS, `BGProcessingTask` hourly catch-up.

Cross-device push uses raw CloudKit (NOT SwiftData) for low latency:
1. Sender writes `CK_OutboxItem` to `OutboxZone`.
2. Every device has a `CKQuerySubscription` filtered by `originDeviceId != self`.
3. Silent push (`content-available: 1`) wakes other devices; `PushHandler` fetches the record, materializes a local `Message`, fires display.

Conflict resolution: CloudKit last-writer-wins per attribute. Acceptable for v1; merge UI deferred.

### 4. CLI ↔ app IPC under sandbox (the trickiest piece)

**Chosen**: App Group container + file-drop queue + Darwin notification kick.

- CLI writes `<unix-ms>-<uuid>.json` to `<AppGroup>/IPC/inbox/`, then `notify_post("com.gavin.khan.ipc.kick")`.
- App registers `CFNotificationCenterAddObserver` + an `FSEventStream` on the inbox dir.
- App validates HMAC, dedupes by UUID, atomically renames into `IPC/processed/`, hands to router.
- Replies (e.g. for `khan inbox tail`) go via `<AppGroup>/IPC/outbox/<request-id>.json`.

Auth: 32-byte HMAC secret in app's Keychain with `kSecAttrAccessGroup = "<TEAMID>group.com.gavin.khan.shared"` so the same-team-signed CLI can read it without prompts. Each request is HMAC-SHA256-signed over canonical JSON. Belt-and-suspenders on top of the team-scoped App Group container.

App-not-running: CLI launches app via `LSOpenCFURLRef` by default (`--no-launch` to skip). On startup the app `drainInbox()`s before showing UI. Cross-device pushes require app running (CLI exits 75 `EX_TEMPFAIL` if `--no-launch` and app dead).

**Rejected alternatives**: loopback HTTP (`com.apple.security.network.server` adds App Store review surface for no benefit); XPC (CLI can't easily resolve bundle path under sandbox); LaunchAgent helper (privileged install dance).

### 5. Mac UI infrastructure

- **Notch panel**: `DynamicNotchKit` Swift Package. Idle pill shows unread count; expand → SwiftUI `NotchExpandedView` with Inbox/Notes/Today tabs. Auto-fallback (floating popup) on non-notch Macs is kept as-is.
- **Sidebar**: custom `KhanSidebarPanel: NSPanel` with `[.borderless, .nonactivatingPanel]`, `level = .floating` normally / `.statusBar - 1` in Fix mode, `collectionBehavior = [.canJoinAllSpaces, .fullScreenAuxiliary, .stationary]` so it survives full-screen apps and space switches.
- **Open Bar**: 6px-wide always-visible trigger panel pinned to the chosen edge.
- **Hot Side**: `NSEvent.addGlobalMonitorForEvents(matching: .mouseMoved)` with 150 ms dwell at the edge. Disabled while frontmost app is full-screen game (configurable blocklist).
- **Global shortcut**: `KeyboardShortcuts` (Sindre Sorhus). Default ⌥⌘K toggles sidebar.
- **Main window**: a regular SwiftUI `Window` scene opened from "Open in window" button.

### 6. Notification routing

```
[external script] → khan-cli
   → <AppGroup>/IPC/inbox/*.json + Darwin notify
   → IPCInboxReader (HMAC verify, dedup)
   → NotificationRouter (mute rules, source allowlist)
       ├─ persist as Message (always) → CloudKit MessagesZone
       ├─ banner mode → BannerPresenter (notch on Mac, UN fallback if app unfocused)
       ├─ fix mode   → NotchPresenter (Mac) / pinned section (iOS)
       └─ broadcast=allDevices/device(...) → OutboxPublisher → CloudKit OutboxZone → silent push to peers
```

Source-specific styling: `claudeCode` (Claude glyph + "Open repo" if `clickAction.openURL` is `file://`), `cliGeneric` (mono title), `scheduledJob` (clock badge), `userMemo`, `share`. Per-source mute rules via `UserSettings.muteRules` (each rule = sourceAppId glob + displayMode filter + silentUntil date).

### 7. CLI surface

`khan <subcmd>` built on Swift `ArgumentParser`:

- `khan notify --title --body --icon --mode banner|fix --source --app-id --click-url --click-note --to <device> | --all-devices --no-launch --json --quiet`
- `khan note add | ls | show | edit | rm`
- `khan inbox ls | tail | dismiss <id> | done <id>`
- `khan push --to <device> ...` (alias for `notify --to ... --mode fix`)
- `khan devices ls`
- `khan auth init | status | path` (HMAC secret rotation; `auth init` requires user confirm in app)
- `khan sync` (force CloudKit poke)
- `khan install --to /usr/local/bin/khan` (symlink bundled binary; offered by first-launch wizard)

Exit codes: `0` ok, `64` usage, `74` IO, `75` temp-fail (app not running + `--no-launch`), `77` permission (HMAC fail).

### 8. Share Extension, App Intents, AppleScript, URL scheme

- **Share Extension** (Mac+iOS): same `<AppGroup>/IPC/inbox/` drop path → main app picks up via FSEvents.
- **App Intents** (`KhanIntents`): `AddNoteIntent`, `PushNotificationIntent`, `MarkMessageDoneIntent`, `OpenSidebarIntent` (Mac only — opens via `khan://` URL because App Intents can't drive AppKit panels directly).
- **AppleScript** (Mac): ships `Khan.sdef` with `make new note` / `push notification` / `dismiss message` commands handled by `NSScriptCommand` subclasses, funneled into the same router.
- **URL scheme**: `khan://note/<id>`, `khan://inbox`, `khan://notify?title=...`. Used as click-action targets.

### 9. Settings (single `UserSettings` row)

`sidebarEdge` (.left/.right, default .right), `sidebarWidth` (320), `hotSideEnabled` (true), `hotSideDwellMs` (150), `openBarVisible` (true), `notchBehavior` (.idlePillVisible), `globalShortcut` (⌥⌘K), `syncPokeIntervalSec` (60), `cloudKitEnabled` (true), `defaultNoteFolder`, `notificationMuteRules`, `cliSourceAllowlist` (default `["claude-code","*"]`), `theme`, `showHexColorPreview` (true), `autoBackupEnabled` (true), `pinnedAcrossSpaces` (true), `firstLaunchCompleted`, `cliInstalledAt`, `deviceName` (default = hostname).

### 10. Sandbox entitlements (macOS)

```
com.apple.security.app-sandbox = YES
com.apple.security.application-groups = ["group.com.gavin.khan.shared"]
com.apple.security.files.user-selected.read-write
com.apple.security.files.bookmarks.app-scope
com.apple.security.network.client                     # CloudKit
com.apple.developer.icloud-services = ["CloudKit"]
com.apple.developer.icloud-container-identifiers = ["iCloud.com.gavin.khan"]
com.apple.developer.aps-environment = "production"    # silent push for cross-device
keychain-access-groups = ["$(TeamIdentifierPrefix)group.com.gavin.khan.shared"]
```

Explicitly **not** requested: `com.apple.security.network.server` (file-drop IPC instead), `com.apple.security.cs.disable-library-validation`, audio/camera, Apple Events outbound. Smaller surface = easier MAS review.

iOS adds Background Modes `remote-notification` + `processing`. CLI binary is signed by the same team and ships only the App Group + Keychain access group entitlements.

### 11. Open risks / deferred

- Windows/Linux clients (CloudKit unavailable) — out of scope.
- Multi-user shared notes via CloudKit Shared DB — v2.
- E2EE beyond CloudKit's own — v2.
- App Store review on Hot Side + always-on-top + global mouse monitor — Bartender/SideNotes have shipped these; have a written explanation ready.
- `/usr/local/bin` write may need sudo on first creation; wizard fallback to `~/.local/bin/khan` + PATH instruction.
- CloudKit rate limits (~40 req/sec/container) under minute poking × N devices × N zones — back off on `CKError.requestRateLimited`.
- DynamicNotchKit external-display behavior on a notched Mac with non-notched external — verify, not a v1 blocker.

## Build / verification plan

End-to-end manual checks (real iOS device required for the iOS-specific ones — CloudKit silent push doesn't work in iOS Simulator):

**A. Bootstrap.** `scripts/generate-project.sh` produces the Xcode workspace. `xcodebuild -workspace Khan.xcworkspace -scheme Khan-macOS build` succeeds.

**B. SwiftData + CloudKit baseline.** Sign into iCloud on Mac+iPad (same account). Create note on Mac → ≤60 s later visible on iPad. Edit on iPad → round-trips. `Logs/khan-debug.log` shows "sync poke fired" each minute. "Sync now" forces an immediate fetch.

**C. CLI banner.** `khan notify --title hi --body world --mode banner` → notch banner on notch'd Mac, floating popup on others. Auto-dismiss after ~5 s. Inbox tab persistently shows the message. Log: HMAC verified, router accepted.

**D. CLI fix mode.** `khan notify --title "build done" --mode fix --click-url file:///path/to/repo` → notch panel expands and stays. Click → Finder opens path. Manual dismiss removes from pinned region but keeps in Inbox.

**E. App-not-running.** Quit app, run `khan notify --title queued`. App launches, drains queue within ~3 s. Repeat with `--no-launch` while quit → file persists, surfaces on next launch.

**F. Cross-device push.** Both Mac+iPad running, same iCloud. `khan notify --all-devices --title test --mode fix` on Mac. iPad receives silent push within ~5 s; PushHandler materializes Message; system local notification fires; badge updates. Open iPad → message in Inbox with `originDeviceId == Mac.deviceId`.

**G. Hot Side + Fix + full-screen.** Enable Fix mode. Open Safari full-screen. Mouse to right edge → sidebar slides in over Safari without stealing focus. Type into a note → sidebar accepts text, Safari stays key in background. Switch spaces → sidebar follows.

**H. Open Bar.** 6 px trigger visible at chosen edge across all spaces. Right-click context menu shows "Move to other edge" / "Hide". Hide persists across relaunch.

**I. Share Extension.** From Safari, share URL via Khan extension → choose "Save as Note" or "Send to Inbox" → matching record appears.

**J. App Intents.** Build a 1-step Shortcut "Push Notification" with title "lunch" → fires identical to CLI path. Repeat on iOS.

**K. AppleScript.** `tell application "Khan" to push notification {title:"hi", body:"applescript", mode:"banner"}` fires. `make new note with properties {title:"foo"}` creates a note.

**L. Backup + restore.** Manual backup → snapshot in `Backups/YYYYMMDD/`. Restore from snapshot UI → notes return.

**M. CLI auth rotation.** `khan auth status` ok. `khan auth init` rotates secret (with in-app confirm). Tampering: hand-edit a queued JSON's body before pickup → router logs HMAC failure and discards.

**N. Sandbox audit.** `codesign -d --entitlements - Khan.app` shows only the entitlements listed in §10. Confirm `com.apple.security.network.server` is **not** present. `xcrun altool --validate-app` against the archive passes.

**O. Performance smoke.** `for i in {1..1000}; do khan notify --title "n$i"; done` → app stays responsive; inbox virtualizes; DedupCache prevents replay duplicates.
